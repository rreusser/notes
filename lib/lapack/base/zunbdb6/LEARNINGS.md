# zunbdb6: Translation Learnings

## Translation pitfalls

- The Fortran `LWORK` parameter is dropped in JS (workspace is
  caller-supplied with stride/offset). Drop it from the wrapper and
  ndarray API to stay consistent with other modules.
- `DLAMCH('Precision')` (used here) is *not* `DLAMCH('Epsilon')`.
  `Precision = Epsilon * Base = 2^-52 = Number.EPSILON` (vs.
  `Epsilon = 2^-53 = Number.EPSILON / 2`). The convergence test reads
  `'Precision'` so use `@stdlib/constants/float64/eps` directly.
- The Fortran reference's first ZGEMV call is conditional on `M1 == 0`
  to avoid touching Q1 with leading dimension `MAX(1, M1) = 1` when
  `M1=0` (the LAPACK ZGEMV would otherwise zero WORK using the empty
  product, which is a no-op, but the M1=0 branch makes the intent
  explicit and avoids passing `LDQ1=0`). Preserved that branch in the
  JS `project()` helper.
- The Fortran second-pass redundantly zeroes WORK before the m1=0
  branch zeros it again; the project() helper inlines the m1=0 zero so
  no outer redundant zero loop is needed in JS.

## Dependency interface surprises

- `zlassq(N, x, stride, offset, scale, sumsq)` returns
  `{scl, sumsq}` — chain two calls for `[X1; X2]` by passing the
  first result's `scl`/`sumsq` into the second. Identical pattern to
  `dlassq` and the same `norm2()` helper structure carries over.
- `zgemv(trans, M, N, alpha, A, sa1, sa2, oa, x, sx, ox, beta, y, sy, oy)`:
  `alpha` and `beta` are `Complex128` scalars (not pairs of doubles or
  `Float64Array(2)`). Hoisted `CONE`, `CZERO`, `CNEGONE` constants at
  module scope to avoid per-call allocation.
- The complex twist vs `dorbdb6`: the projection coefficient pass uses
  `'conjugate-transpose'` (Q^H) rather than `'transpose'` (Q^T). The
  back-projection still uses `'no-transpose'`. Real-Q tests pass
  identically, but a complex Q test (case `complex_q_one_col`) is
  needed to actually exercise the conjugation.

## Complex number handling

- `Complex128Array` strides/offsets are in **complex elements** at the
  zgemv/zlassq API. For the manual zero-fill helper, multiply by 2 to
  index a `Float64Array` view (`stride * 2`, `offset * 2`).
- Hoisted `X1v = reinterpret(X1, 0)` etc. once at the top of `zunbdb6`
  to amortize the cost across the (up to two) zero-fill calls.
- Avoided naming any local Float64 view variable in the `zero()`
  helper after the regex `(Av|Bv|Cv|Hv|Zv|Qv|Tv|Uv|Xv|VTv|VLv|VRv)`
  — the `stdlib/z-prefix-reinterpret` rule treats those names as
  "complex array indices needing reinterpret" and flags helpers that
  receive an already-reinterpreted Float64Array. Renamed the helper's
  parameter to `buf` to satisfy the rule without adding a no-op
  reinterpret call.

## Fortran test

- Used EQUIVALENCE on 1D X1/X2 arrays to print interleaved real/imag
  doubles. Did NOT EQUIVALENCE Q1/Q2 (2D) — the LDA=NMAX vs N
  mismatch would produce garbage. We don't print Q in the fixture
  (only X1/X2 outputs), so this avoids the matrix-print pitfall
  entirely.
- Added a 9th test case `complex_q_one_col` with a genuinely complex
  Q (entries `(0.5, ±0.5i)`) — required to actually exercise the
  conjugate-transpose path. Without it, all-real Q tests would pass
  even with a `'transpose'` bug.

## Process improvements

- The scaffolded `signature.py` again emitted both
  `strideX1, offsetX1, incx1` etc. for arrays with `INCX`-style
  parameters. Same fix as `dorbdb6`: drop the redundant `incx*`/`lwork`
  slots when writing the actual base/wrapper signatures.
- Same `init_routine.py` arity-mismatch in `test/test.<routine>.js` as
  `dorbdb6` (counted 16 instead of the wrapper's 14). Re-derived
  expected arity from the wrapper.
- The deps file generated by `init_routine.py` was missing
  `la_constants`, `la_xisnan`, `dlamch` (transitive .f90 module deps
  for `zlassq`). Pattern is identical to `dorbdb6` — worth teaching
  `init_routine.py` (or `deps.py`) to auto-add these whenever a norm
  routine like `*lassq` is in the tree.
