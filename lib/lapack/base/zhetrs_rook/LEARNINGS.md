# zhetrs_rook: Translation Learnings

## Translation pitfalls

- Structurally identical to non-rook `zhetrs`. The ONLY difference is that
  rook pivoting performs TWO row swaps per 2x2 pivot block (one for row
  k, one for row k+/-1) and BOTH `IPIV` entries of the block are
  negative. The non-rook variant uses a single negative entry.
- The forward sweep (UDU^H solve / LDL^H solve) and the back sweep
  (U^H solve / L^H solve) both apply the swaps; the back sweep applies
  them in the opposite order to mirror the factorization sequence.
- Hermitian: `A[k,k]` is real, so the 1x1 pivot uses only `Av[diagP]`
  (the real part) for `1/D(k,k)`. Off-diagonal `AKM1K` is complex but
  the diagonal usage in the 2x2 D solve treats `A(k-1,k-1)`/`A(k,k)`
  as real (passing zero as imaginary).
- The `U^H * X` sweep wraps `zgemv('conjugate-transpose', ...)` with a
  pair of `zlacgv` calls on the target B row to flip its imaginary sign.
  This pattern must be replicated exactly for both rows of a 2x2 block.

## Fortran IPIV → JS conversion

- Fortran 1-based positive `p` → JS 0-based `p - 1`.
- Fortran negative `-p` → JS `~(p-1) = -p` (numerically identical).
  No transformation needed for negatives.

## Fixture strategy

- The Fortran test prints `lda` (NMAX), the factored `A` (lda*n
  doubles, interleaved re/im), `IPIV`, the input `b`, and the solved
  `x`. The JS test reconstructs the column-major matrices using the
  printed `lda` (not `n`) as the stride.
- The Fortran test deps file needed `zhetrf_rook`, `zhetf2_rook`,
  `zlahef_rook`, `zlacgv`, `dlamch`, `dlapy2`, `disnan`, `dlaisnan`,
  `ilaenv`, `ieeeck`, `iparmq` in addition to the autogenerated
  `zhetrs_rook`. `deps.py` only sees the dependencies of
  `zhetrs_rook` itself, not those of `zhetrf_rook` invoked by the test.

## Complex arithmetic

- The 2x2 D solve uses inlined `cDiv` (Smith algorithm) stored in
  module-level `cdR`/`cdI` to avoid allocation in the per-RHS loop.
  Pattern lifted from the existing `zhetrs` module.
