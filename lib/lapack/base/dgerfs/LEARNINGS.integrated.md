# dgerfs: Translation Learnings

## Translation pitfalls

- The Fortran WORK array is segmented into three blocks of N doubles: WORK(1..N) for abs values, WORK(N+1..2N) for residual/RHS, WORK(2N+1..3N) for dlacn2's v vector. In JS, these map to indices 0..N-1, N..2N-1, 2N..3N-1. Getting the offsets right when passing to dcopy/dgemv/dgetrs was the main translation concern.
- The backward GOTO at label 20 (iterative refinement loop) maps cleanly to a `while(true)` with a `break` at the stopping criterion.
- The backward GOTO at label 100 (dlacn2 reverse communication loop) similarly maps to a `while(true)` with `break` when KASE=0.
- WORK and IWORK are allocated internally (not passed as parameters) since they are pure scratch space. This simplifies the API vs the Fortran version.

## Dependency interface surprises

- dlacn2 uses a reverse-communication pattern with KASE and EST as single-element arrays (Int32Array and Float64Array respectively), plus ISAVE as a 3-element Int32Array for state persistence across calls. The ISGN parameter (called IWORK in the Fortran dgerfs) serves as dlacn2's sign array.
- dgetrs and dgemv both use full-word trans strings ('no-transpose', 'transpose') rather than single chars ('N', 'T'). The Fortran TRANST variable ('T' when NOTRAN, 'N' otherwise) maps to 'transpose'/'no-transpose' in JS.

## Automation opportunities

- The deps file generated by init_routine.py only includes direct dependencies of dgerfs itself. For Fortran test compilation, transitive deps (dgetrf, dgetrf2, ilaenv, ieeeck) had to be added manually to deps_dgerfs.txt. A recursive dep resolution for the test deps file would save manual work.

## Coverage gaps

- The SAFE2 branches (lines 173-174, 180-188, 202-203 in base.js) handle near-underflow inputs where WORK[i] <= SAFE2. These require inputs with magnitudes near 1e-308 to trigger. Coverage is at 94.78% line / 84.85% branch without targeting these paths, which is above threshold.
