# zspsvx: Translation Learnings

## Translation pitfalls

- The Fortran RCOND is a scalar output. In JS, it must be a Float64Array(1) so the callee can write into it. This matches the dspsvx pattern.
- zspsvx uses packed storage (1D Complex128Arrays AP, AFP) -- only stride and offset, no stride1/stride2.
- The signature conformance linter warns about WORK/RWORK params (6 extra params vs the Fortran-implied 28). This is expected and matches the dspsvx pattern.
- Unlike dspsvx which uses WORK (Float64Array) + IWORK (Int32Array), zspsvx uses WORK (Complex128Array) + RWORK (Float64Array). The workspace types differ between real and complex variants.

## Dependency interface surprises

- zspcon writes rcond into rcond[0] (Float64Array) rather than returning it, same as dspcon.
- zsptrf returns info (integer), not an object.
- zsprfs takes WORK/RWORK params but allocates internally; the params are pass-through for API compatibility.
- zlansp uses RWORK (Float64Array) for its workspace, not WORK (Complex128Array). The RWORK parameter from zspsvx is passed through to zlansp for its workspace.
- All z* dependencies use complex-element strides and offsets (not Float64 element indices). The reinterpret pattern is handled inside each dependency.

## Automation opportunities

- The `deps_*.txt` file generated by `deps.py` missed `la_xisnan` and `iparam2stage` which are needed for Fortran compilation of routines transitively using zlassq (via zlansp). This is a known limitation documented in CLAUDE.md.

## Coverage gaps

- No test for negative-stride or offset-based access since this is a driver routine and the base tests use stride=1, offset=0 throughout.
- The ill-conditioned test from dspsvx is omitted because complex symmetric Hilbert-like matrices have different conditioning behavior.

## Complex number handling

- All packed arrays (AP, AFP), matrix arrays (B, X), and workspace WORK are Complex128Array. FERR, BERR, RWORK, and rcond are Float64Array.
- The base.js itself does no reinterpret calls -- it delegates entirely to dependencies (zcopy, zlacpy, zlansp, zsptrf, zsptrs, zspcon, zsprfs) which handle the Complex128Array <-> Float64Array view internally.
- Fortran test uses EQUIVALENCE to alias complex arrays to double precision arrays for JSONL output. The fixture data stores interleaved re/im pairs.
