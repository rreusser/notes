# dtfttr: Translation Learnings

## Translation pitfalls

- Fortran uses 0-based arrays (`ARF(0:*)`, `A(0:LDA-1, 0:*)`), which is unusual and makes the index translation to JS straightforward -- no off-by-one adjustments needed for loop bounds.
- The `lda` parameter is consumed entirely into `strideA1`/`strideA2` and is unused in `base.js`. Added `eslint-disable no-unused-vars` on the function line.
- The Fortran uses `NX2 = N + N` and `NP1X2 = N + N + 2` as stride-back offsets for the reverse-traversal loops in the upper-triangle normal-format paths. These must be computed conditionally based on `nisodd` and `lower`, exactly matching the Fortran structure.
- The `j = k-1` exit value from the `DO J = 0, K-2` loop is used after the loop in the even/transpose/upper path. This is a Fortran idiom where the loop variable retains its last value. The JS translation must preserve `j` in scope for the trailing loop.

## Dependency interface surprises

- N/A: dtfttr has no BLAS/LAPACK dependencies (pure index arithmetic).

## Automation opportunities

- The `transr` parameter validation was not auto-generated by `init_routine.py`. Had to manually add `isTransposeOperation` validation to `ndarray.js`. The scaffold generator could detect TRANSR-type params and add this automatically.

- No coverage gaps: 100% line, branch, and function coverage achieved. All 8 code paths (N odd/even x normal/transpose x lower/upper) plus edge cases (N=0, N=1) and offset/LDA tests are covered.
