# dsysv: Translation Learnings

## Translation pitfalls

- dsysv is a simple driver routine (factorize + solve), so the translation is straightforward. The main pitfall was the IPIV encoding mismatch: dsytrf uses bitwise NOT `~kp` for 2x2 pivot indices (0-based), while Fortran uses negative 1-based values. The mapping is: Fortran `-k` equals JS `~(k-1) = -k`, so negative IPIV values are numerically identical between Fortran and JS. Only positive IPIV values need the standard `-1` adjustment for 0-based conversion.
- The Fortran dsysv has a LWORK/workspace query path and conditionally calls dsytrs vs dsytrs2 depending on LWORK. In JS, we skip the workspace query entirely, allocate internally, and always use dsytrs2 (the BLAS-3 variant), which is the preferred path.

## Dependency interface surprises

- dsytrf expects `'upper'`/`'lower'` string arguments (full words), not the single-char `'U'`/`'L'` convention documented in CLAUDE.md. This is inconsistent with dsytrs2, which uses `'U'`/`'L'`. The dsysv driver must map between these conventions when calling each dependency.
- dsytrf does not take WORK/LWORK parameters — it allocates workspace internally. This simplifies the driver but differs from the Fortran interface.

## Automation opportunities

- The deps file generated by `init_routine.py` for LAPACK routines is missing infrastructure dependencies like `ilaenv`, `ieeeck`, `iparmq`, and `dlamch` that are needed by `dsytrf` at the Fortran level. These had to be manually added to `deps_dsysv.txt` for the Fortran test to compile. The `deps.py` script should include these transitive infrastructure dependencies.

## Coverage gaps

- 100% line, branch, and function coverage on base.js. No gaps — the driver is simple enough that all paths are covered by the 9 test cases.
