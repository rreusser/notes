# dsytrs_rook: Translation Learnings

## Translation pitfalls

- The structural skeleton is identical to non-rook `dsytrs`. The ONLY
  semantic difference is the 2x2 pivot block: rook pivoting performs TWO
  row interchanges per 2x2 block (one for row k, one for row k+/-1),
  whereas the regular Bunch-Kaufman variant performs only ONE.
- For rook 2x2 blocks, BOTH `IPIV[k]` and `IPIV[k+/-1]` are negative.
  The non-rook variant typically encodes the pair using a single negative
  entry. When porting, audit every IPIV access — the second-row swap
  reads `IPIV[k-1]` (upper) or `IPIV[k+1]` (lower).
- The forward sweep (U/D solve, L/D solve) and the back sweep
  (U^T solve, L^T solve) both need the two swaps, applied in opposite
  orders to mirror the factorization sequence.

## Fortran IPIV → JS conversion

- Fortran 1-based positive `p` → JS 0-based `p - 1`.
- Fortran negative `-p` → JS `~(p-1) = -p` (numerically identical, no
  conversion needed). The `convertIpiv` helper in the test file embodies
  this rule.

## Fixture strategy

- Because no JS `dsytrf_rook` exists yet, the Fortran test prints both the
  factored matrix `A` and `IPIV` in addition to the solved `B`. The JS
  test reconstructs `A` and `IPIV` directly from the fixture, then calls
  only `dsytrs_rook`. This isolates the solve-only logic from the (not
  yet ported) rook factorization.
- The Fortran test deps file needed `dlamch`, `dlasyf_rook`, `dsytf2_rook`,
  `dsytrf_rook` in addition to the autogenerated `dsytrs_rook`. `deps.py`
  did not pull these because they are referenced only by the test program,
  not by `dsytrs_rook` itself.
