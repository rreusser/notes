# dspsvx: Translation Learnings

## Translation pitfalls

- The Fortran RCOND is a scalar output. In JS, it must be a Float64Array(1) so the callee can write into it. This matches the dsysvx/dposvx pattern.
- dspsvx uses packed storage (1D arrays AP, AFP) unlike dsysvx which uses 2D arrays. No stride1/stride2 for the packed arrays -- just stride and offset.
- The signature conformance linter warns about WORK/IWORK params (6 extra params vs the Fortran-implied 28). This is expected and matches the dsysvx pattern.

## Dependency interface surprises

- dspcon writes rcond into rcond[0] (Float64Array) rather than returning it.
- dsptrf returns info (integer), not an object.
- dsprfs takes WORK/IWORK params but allocates internally; the params are pass-through for API compatibility.

## Automation opportunities

- The `deps_*.txt` file generated by `deps.py` missed `la_constants`, `la_xisnan`, and `iparam2stage` which are needed for Fortran compilation of routines transitively using dlassq (via dlansp). This is a known limitation documented in CLAUDE.md.

## Coverage gaps

- No test for the lwork query path (dsysvx supports it but dspsvx Fortran does not have a workspace query -- it uses fixed 3*N workspace).
- No test for negative-stride or offset-based access since this is a driver routine and the base tests use stride=1, offset=0 throughout.

## Complex number handling

- N/A: dspsvx is a real-valued routine.
