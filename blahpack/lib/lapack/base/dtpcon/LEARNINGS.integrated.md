# dtpcon: Translation Learnings

## Translation pitfalls

- dtpcon is structurally identical to dtrcon but uses packed storage (1D array with stride) instead of 2D full storage (strideA1, strideA2). The key difference is calling dlantp/dlatps instead of dlantr/dlatrs. No index arithmetic changes needed in the reverse-communication loop itself.
- The deps file generated by `deps.py` was missing `la_constants` and `la_xisnan`, which are Fortran module dependencies required by `dlassq.f90` for compilation. Added them manually.

## Dependency interface surprises

- dlatps uses the same packed-storage interface as dlantp: (uplo, trans, diag, normin, N, AP, strideAP, offsetAP, x, strideX, offsetX, scale, CNORM, strideCNORM, offsetCNORM). The CNORM workspace lives at WORK[2N..3N-1], same layout as dtrcon.
- dlacn2 reverse communication loop is identical between dtrcon and dtpcon -- no packed-storage awareness needed since it only operates on the x and v workspace vectors.

## Automation opportunities

- The packed-storage condition number routines (dtpcon, dppcon) follow an extremely regular pattern. A template could generate them from the full-storage versions by swapping the norm/solve routines and adjusting the matrix parameter from (A, stride1, stride2, offset) to (AP, stride, offset).

## Coverage gaps

- The early-exit path where `scale < xnorm * smlnum` (overflow guard in the SCALE loop) is not tested. This requires a near-singular matrix that triggers the careful solve path in dlatps with extremely small scale factors, which is difficult to construct.

## Complex number handling

- N/A: dtpcon is a real-valued routine.
