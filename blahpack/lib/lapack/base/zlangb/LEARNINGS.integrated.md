# zlangb: Translation Learnings

## Translation pitfalls

- Fortran band indexing `AB(KU+2-J, J)` uses 1-based indices. Converting to 0-based: `i_start = max(KU+1-j, 1) - 1` and `i_end = min(N+KU-j, KL+KU+1)` (exclusive).
- The infinity norm branch uses logical matrix row indices `I=MAX(1,J-KU)..MIN(N,J+KL)` (Fortran 1-indexed) with band row offset `K=KU+1-J`, requiring careful conversion to 0-based: `k = KU - j`, `i` from `max(0, j-KU)` to `min(N, j+KL+1)` (exclusive).
- Frobenius norm: Fortran `L = MAX(1, J-KU)` and `K = KU+1-J+L` compute the starting band row. In 0-based: `l = max(0, j-KU)`, `k = KU - j + l`, count = `min(N, j+KL+1) - l`.

## Dependency interface surprises

- `zlassq` returns `{scl, sumsq}` (not `{scale, sum}`). Must use `out.scl` and `out.sumsq`.
- `zlassq` takes complex element strides/offsets, not Float64 indices. Pass `strideAB1` and `offsetAB + k*strideAB1 + j*strideAB2` directly.

## Automation opportunities

- The deps_zlangb.txt auto-generated by `deps.py` missed `la_constants` and `la_xisnan` (module dependencies of `zlassq.f90`). Had to copy from `deps_zlantb.txt`.

## Coverage gaps

- All four norm types tested with two different band widths (5x5 with KL=1/KU=2 and 4x4 tridiagonal KL=1/KU=1) plus diagonal-only (KL=0/KU=0) and 1x1 edge cases. N=0 quick return tested.

## Complex number handling

- Uses `cmplx.absAt()` for element-wise absolute values in max/one-norm/inf-norm branches.
- Frobenius norm delegates to `zlassq` which handles complex sum-of-squares internally (processes real and imaginary parts separately with Blue's scaling constants).
- No inline complex arithmetic needed -- zlangb only computes real norms of complex data.
